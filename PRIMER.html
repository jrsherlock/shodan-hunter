01 / THE BRIEF

What it is, who it's for

One shared Shodan key, many analysts, every query attributed. It turns one-shot recon into a team-friendly, auditable, cost-aware workspace.

— mission.md

The audience: a security / recon team — pentesters, threat analysts, blue-teamers — who share a single Shodan Small-Business key and need to see each other's work without each holding the key.

The job: ask "what does the internet know about this org / domain / netblock?" in plain English, get a clean answer, and drill into any host — while a soft daily budget and an audit log keep one shared key from quietly draining.

What it isn't (yet): an autonomous investigator. The LLM writes one query at a time — it doesn't chain calls or summarize results. Basic auth is the door, not an audit boundary; put it behind SSO / a reverse proxy for anything beyond the LAN.

04 / ENABLING THE GATED FEATURES

Those three .env lines, decoded

“Gated” = high-impact features kept off (or guarded) until you deliberately opt in. You set these as environment variables in your .env file, then restart. Here's exactly what each line does.

— .env
# add these, then restart the app
SH_ENABLE_SCAN=11
SH_SCAN_ALLOWLIST=203.0.113.0/24,198.51.100.0/242
# SH_ENABLE_ALERTS=1  ← already the default3
  1. The master switch for scanning. It's false by default. Set it to 1 and /scan flips from a “disabled” notice into a working form — and POST /scan starts running instead of returning 403.
  2. Your authorized ranges. A scan is rejected unless every target sits inside one of these CIDRs. Leave it empty and the form instead forces an explicit “I am authorized” checkbox on each scan (still logged). Cap per scan: SH_SCAN_MAX_HOSTS (default 4096).
  3. Not active config — it's a comment. The leading # means this line does nothing; it's a reminder that alerts are already on. To turn alerts off, set SH_ENABLE_ALERTS=0 (uncommented).
⚠️

Those example IPs are placeholders. 203.0.113.0/24 and 198.51.100.0/24 are RFC 5737 documentation ranges (TEST-NET-3 / TEST-NET-2) — they exist precisely so they're never real targets. Replace them with netblocks you actually own or are contracted to scan. Pointing Shodan's crawlers at infrastructure you're not authorized for is your liability, on your key.

05 / A SESSION

Start to finding

— playbook

  1. Sign in

    The browser prompts for HTTP basic auth. Each teammate has their own username so the log can name them.

  2. Ask in English

    “Find exposed RDP on hosts owned by org Acme Industries”

    Azure OpenAI returns org:"Acme Industries" port:3389 — editable, so you refine without retyping.

  3. Read the room

    Results land with free InternetDB enrichment, 🍯 honeypot badges on decoys, and a facet strip (top ports / countries / orgs) above the table — all at 0 extra credit.

  4. Drill in

    Click any IP for the full host record at /host/{ip}; pivot to /domain for subdomains or /dns for a bulk resolve.

  5. Watch it over time

    Register the client's netblocks under /alerts so changes trip a trigger — or, once enabled, kick a fresh /scan of a range you're cleared for.

  6. Leave a trail

    Prompt, generated query, result count, your name, the action and the credits spent all append to /log.

06 / GUARDRAILS

Built to not surprise you

▣Per-user auth

Every request is attributed; the log shows who ran what.

▣Soft daily budget

SH_DAILY_BUDGET caps team query credits per UTC day.

▣Scan authorization

Allowlist or explicit confirm, host-capped, off by default.

▣Free-first design

InternetDB + count + DNS run before you ever spend a credit.
